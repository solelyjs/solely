🛡️ Solely 框架安全机制演示

演示目的：展示 Solely 框架如何处理潜在的安全风险，特别是 XSS（跨站脚本攻击）。 框架的模板在编译时就已经确定，运行时只能修改数据，不能改变模板结构。

🚨 场景 1: 用户资料中的 XSS 攻击

攻击者尝试在用户资料中注入恶意代码，看看框架如何处理。

⚠️ 重要安全警告

本示例使用了 :innerHTML、:href 和 :src 指令来展示 XSS 风险。在实际项目中：

  • 永远不要将用户输入的内容直接绑定到 :innerHTML
  • 永远不要将第三方 API 数据直接绑定到 :innerHTML
  • 永远不要将用户输入的 URL 直接绑定到 :href
  • 恶意 URL 如 javascript:alert('XSS') 会在点击时执行代码！
  • :src 绑定 javascript: 协议在现代浏览器中不会执行，但仍建议过滤
  • :innerHTML 和 :href 应该只用于完全可信的静态内容
  • 使用 {{ }} 插值会自动转义 HTML，是安全的选择

攻击数据示例：

{ username: '<img src=x onerror="alert(\'XSS\')">', bio: '<script>alert("XSS")</script>正常简介', website: 'javascript:alert("XSS")', userContent: '<div onclick="alert(\'点击攻击\')">点击我</div>', avatarUrl: 'javascript:alert("XSS via img")' }

⚠️ 场景 2: 评论中的 XSS 攻击

攻击者在评论中注入恶意代码，测试框架的防护能力。

框架的安全机制：

  • {{ }} 插值表达式会自动进行 HTML 实体编码
  • <script> 标签会被转义为纯文本显示
  • 事件处理器（如 onerror）不会被绑定

ℹ️ 场景 3: URL 参数注入尝试

尝试通过 URL 参数注入恶意模板代码。

为什么这种攻击无效：

  • Solely 的模板在编译时就已经确定
  • 运行时只能修改 $data 中的数据
  • URL 参数无法改变已编译的模板结构
  • 即使传入 {{ alert('XSS') }}，也会被当作纯文本

✅ 场景 4: Class 绑定特殊字符测试

测试框架如何处理包含特殊字符的 class 名称，以及验证 class 绑定的安全性。

框架的安全机制：

  • classList.add() 接受字符串作为 class 名称
  • 包含空格的 class 名称会被自动过滤（开发模式下会警告）
  • 不会解析字符串内容为选择器或代码
  • 无法通过 class 名称执行 JavaScript 代码

测试数据示例：

{ // 有效的 class 名称 'has-dash': true, 'has_underscore': true, 'emoji-😀': true, '中文类名': true, // 无效的 class 名称（会被过滤） 'has space': true, // XSS 尝试（不会执行） '<script>alert("XSS")</script>': true, 'onclick=alert("XSS")': true, 'onerror=alert("XSS")': true }

⚠️ 场景 5: 事件处理器属性绑定行为

测试 :onclick 属性绑定的实际行为。Solely 框架通过设置 DOM property 实现绑定，字符串不会被执行。

ℹ️ 框架行为说明

Solely 的 :attr 绑定设置的是 DOM property（el.onclick = value），不是 HTML attribute。

当 value 是字符串时，由于 onclick property 期望的是函数，所以字符串不会被执行。

{ clickHandler: "alert('XSS via onclick: ' + document.cookie)" // 结果：el.onclick 被设置为字符串，但不会执行 }

📘 场景 6: Style 属性绑定行为

测试 :style 属性绑定的行为。由于 Solely 不支持 IE，style 中的 javascript: 协议不会执行。

ℹ️ 框架说明

通过 style 属性执行 XSS（如 background-image: url('javascript:...')）仅在旧版 IE 中有效。

Solely 框架不支持 IE，且现代浏览器已修复此漏洞，因此这种攻击方式在本框架下不存在风险。

{ userStyle: "background-image: url('javascript:alert(\"XSS\")')" // 结果：样式会被应用，但 javascript: 代码不会执行 }

✅ Solely 框架的安全机制总结

🛡️ 自动防护

  • HTML 实体编码
  • 模板编译时固定
  • 数据与结构分离
  • 属性绑定安全检查

⚠️ 需要注意

  • :innerHTML 指令（如果存在）会渲染原始 HTML
  • javascript: 伪协议在 href/src 中
  • 动态模板构造（如果支持）
  • 第三方脚本引入

框架的安全设计：

  1. 编译时安全：模板在构建阶段编译为 IR，运行时不可更改
  2. 自动转义：{{ }} 插值自动进行 HTML 实体编码
  3. 数据隔离：用户数据只能影响内容，不能改变 DOM 结构
  4. 函数预编译：事件处理函数在编译时确定，不是动态解析

⚠️ 开发者安全注意事项

🚫 绝对禁止：使用 :innerHTML 绑定不可信内容

// ❌ 绝对不要这样做！会导致 XSS 攻击 // 用户输入的内容直接绑定到 :innerHTML <div :innerHTML="$data.userComment"></div> // ❌ 绝对不要这样做！API 数据可能已被污染 <div :innerHTML="$data.apiResponse.content"></div> // ❌ 绝对不要这样做！URL 参数不可信 <div :innerHTML="getUrlParam('content')"></div>

后果：攻击者可以注入 <script> 标签、事件处理器（onclick/onerror）等恶意代码， 窃取用户 Cookie、Token，或执行任意操作。

⚠️ 危险：使用 :href / :src 绑定用户输入的 URL

// ❌ 危险：用户输入的 URL 可能包含 javascript: 协议 <a :href="$data.userWebsite">访问网站</a> // ❌ 危险：图片 src 可能被注入恶意协议 <img :src="$data.userAvatar"> // ❌ 危险：不验证的外部链接 <a :href="externalLink">外部链接</a>

后果：攻击者可以使用 javascript:alert('XSS') 等伪协议， 用户点击链接时执行恶意代码，窃取 localStorage、Cookie 等敏感信息。

ℹ️ :style 绑定说明

// 虽然可以绑定包含 javascript: 协议的样式 <div :style="$data.userStyle">样式区域</div> // 用户输入: background-image: url('javascript:alert("XSS")') // 但在 Solely 框架中（不支持 IE），这种攻击无效

说明：通过 CSS 的 url('javascript:...') 执行代码仅在旧版 IE 中有效。 Solely 框架不支持 IE，且现代浏览器已修复此漏洞，因此不存在 XSS 风险。

✅ 正确的做法：使用 @click 绑定事件（而非 :onclick）

// ❌ 不推荐：虽然 Solely 中不会执行，但语义不清晰 <div :onclick="$data.userScript">点击我</div> // ✅ 正确：使用框架提供的 @click 事件绑定 <button @click="handleClick()">点击我</button> // ✅ 正确：事件处理函数在组件中定义，不是用户输入 class MyComponent extends BaseElement { handleClick() { // 安全的处理逻辑 } }

说明：Solely 的 :onclick 绑定设置的是 DOM property，字符串不会被执行。 但仍建议使用框架的 @click 语法，语义更清晰，且完全不受用户输入影响。

其他应该避免的做法：

// ❌ 不要手动构造模板字符串 const template = `<div>${userInput}</div>`; // ❌ 不要直接操作 DOM 插入不可信内容 element.innerHTML = userContent; // ❌ 不要信任外部 API 数据而不验证 const data = await fetch('https://api.example.com/data'); // 直接使用而不验证

✅ 正确的做法：使用 {{ }} 插值（自动转义）

// ✅ 正确：使用 {{ }} 插值，框架会自动转义 HTML // 即使 userComment 包含 <script> 标签，也会被转义为纯文本显示 <div>{{ $data.userComment }}</div> // ✅ 正确：API 数据使用 {{ }} 绑定 <div>{{ $data.apiResponse.content }}</div> // ✅ 正确：如果需要富文本，使用专门的富文本组件并做白名单过滤 <rich-text :content="sanitizedHtml"></rich-text>

✅ 正确的做法：URL 白名单过滤

// ✅ 正确：对 URL 进行协议白名单过滤 function sanitizeUrl(url) { if (!url) return '#'; // 只允许 http/https 协议 if (/^(https?:)?\/\//i.test(url)) { return url; } return '#'; } // ✅ 正确：过滤后再绑定到 :href <a :href="sanitizeUrl($data.userWebsite)">访问网站</a> // ✅ 正确：图片 URL 同样过滤 <img :src="sanitizeUrl($data.userAvatar)">

其他推荐的安全实践：

  • ✅ 对用户输入进行验证和清理（白名单策略）
  • ✅ 使用 CSP (Content Security Policy) 限制脚本执行
  • ✅ 对第三方 API 数据进行 schema 验证
  • ✅ 如果必须使用 :innerHTML，确保内容经过严格的 HTML 净化（如 DOMPurify）
  • ✅ 设置 HttpOnly Cookie，防止 JavaScript 读取敏感 Cookie
  • ✅ 对 :href 绑定的 URL 进行协议白名单过滤（防止 javascript: 协议执行）
  • ✅ 对 :src 绑定的 URL 也建议进行过滤（虽然现代浏览器不会执行 javascript: 协议）
  • ✅ 用户输入的链接显示前，检查并过滤 javascript:, data:, vbscript: 等危险协议
  • ✅ 使用 @click 框架事件绑定语法（预编译的函数），而不是 :onclick（字符串不会执行但语义不清）
